Lesson 4 · Results & storage

Cache big artifacts in object storage

An encoded mp4 is megabytes, not an integer. You don't want to stuff it into Prefect's result store — you want it in object storage, and you want Prefect to remember "I already encoded this, here's where it is" so it never transcodes the same video twice. We'll do it on a local MinIO store, then point the same code at Tencent COS.

Lesson 3 cached a tiny dict on your laptop. Real pipelines produce big outputs — a transcoded video, a trained model, a Parquet export — and run on more than one machine. This lesson's win: encode a video once, store the mp4 in a local S3-compatible object store (MinIO), and watch the second run come back Cached without re-encoding. MinIO speaks S3, so moving to Tencent COS later is a credentials swap — not a rewrite.

1 · The decision that matters: what gets stored?

Prefect caching stores a task's return value in its result store.1 So the question is: what should encode_video return?

ApproachWhat's cachedVerdict
Return the mp4 bytes The whole video, serialized into the result store ✗ Fights the serializer, bloats storage, loses the .mp4 content-type
Upload the mp4 yourself, return its key A short string like encoded/<hash>/1080p.mp4 ✓ The video lives in COS as a real object; Prefect caches a cheap pointer
The mental correction result_storage stores the return value — not your big file. The mp4 lands in COS because your code uploads it there. Prefect just caches the key and skips the re-encode. Keep these two jobs separate in your head and the rest is easy.

2 · Key on content, not filename

Recall from Lesson 3: cache_policy=INPUTS keys on the task's arguments.2 So pass a content hash of the source plus the preset — never just a path. A path-keyed cache hands back a stale encode when the file changes; a content hash busts the cache exactly when the bytes change, and the preset busts it when you re-encode at a new resolution. 

import hashlib
from pathlib import Path
from prefect import task
from prefect.cache_policies import INPUTS

def content_hash(path: Path) -> str:
    return hashlib.sha256(path.read_bytes()).hexdigest()[:12]

@task(cache_policy=INPUTS)
def encode_video(src_hash: str, preset: str = "h264-1080p") -> str:
    out_key = f"encoded/{src_hash}/{preset}.mp4"
    # ... ffmpeg transcode -> upload to COS ...
    return out_key          # the small string Prefect caches

3 · Spin up local object storage (MinIO)

MinIO is an S3-compatible object store you run on your own machine — perfect for learning locally before you touch a cloud account. A docker-compose.yml is in your repo; it starts MinIO and creates the prefect-artifacts bucket: 

docker compose up -d        # MinIO on :9000, web console on :9001

Browse localhost:9001 (login minioadmin / minioadmin) — that's your object store's UI, where the encoded mp4 will appear.

Prefect talks to it through an S3Bucket storage block (from prefect-aws, already installed via the prefect[aws] extra).2 MinIO has its own credentials class: 

from prefect_aws import MinIOCredentials, AwsClientParameters, S3Bucket

cos = S3Bucket(
    bucket_name="prefect-artifacts",
    credentials=MinIOCredentials(
        minio_root_user="minioadmin",
        minio_root_password="minioadmin",
        aws_client_parameters=AwsClientParameters(endpoint_url="http://localhost:9000"),
    ),
)
cos.save("minio-artifacts", overwrite=True)   # save once; later: S3Bucket.load("minio-artifacts")

4 · Run it and watch the cache

04_results_storage.py is in your repo — it builds that block, encodes a (fake) video, uploads it to MinIO, and returns the key. With the Prefect server and MinIO running: 

uv run python 04_results_storage.py     # run it twice

Run 1 prints encoding … (slow), finishes Completed, and the mp4 shows up in the MinIO console under encoded/<hash>/. Run 2 prints no encode line, returns the same key, and the task is Cached — the encode and the upload were skipped across a separate process, because INPUTS persists results between runs (the RUN_ID gotcha from Lesson 3). The task itself is just: 

@task(cache_policy=INPUTS, result_storage=cos)
def encode_video(src_hash, preset="h264-1080p"):
    out_key = f"encoded/{src_hash}/{preset}.mp4"
    local_mp4 = run_ffmpeg(...)                # your transcode -> local file
    cos.upload_from_path(local_mp4, out_key)   # the artifact upload
    return out_key                             # the small string Prefect caches

5 · Same code, real Tencent COS

This is the payoff of S3-compatibility: to go from MinIO to Tencent COS, you change only the block — the task code is untouched. Swap MinIOCredentials for AwsCredentials and point it at the COS endpoint:1 

from prefect_aws import AwsCredentials, AwsClientParameters, S3Bucket

cos = S3Bucket(
    bucket_name="videos-1250000000",          # Tencent: name MUST include your APPID
    credentials=AwsCredentials(
        aws_access_key_id="<SecretId>",        # from Tencent CAM
        aws_secret_access_key="<SecretKey>",
        region_name="ap-guangzhou",
        aws_client_parameters=AwsClientParameters(
            endpoint_url="https://cos.ap-guangzhou.myqcloud.com",
        ),
    ),
)
cos.save("minio-artifacts", overwrite=True)   # reuse the block name; code that loads it doesn't change
Tencent COS specifics (verified against prefect-aws) Three things that trip people up: the bucket name carries your APPID (videos-1250000000, not videos); the endpoint is https://cos.<region>.myqcloud.com; and Tencent's SecretId / SecretKey map onto aws_access_key_id / aws_secret_access_key.

6 · The gotcha before you ship

Prefect decides a cache hit from its own result record, not from whether the stored object still exists. If a lifecycle rule or a teammate deletes the mp4 but the cache record survives, the task "hits" and returns a pointer to a missing file. Two defenses:

  • Guard with an existence check (cos.read_path / a HEAD) and fall through to re-encode if the object is gone.
  • Align cache_expiration with your COS lifecycle policy so they expire together.
