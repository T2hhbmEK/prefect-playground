Lesson 13 · Coordinating distributed runs

Gather the fan-out

L12 fired each encode off fire-and-forget (timeout=0) and walked away. But a real coordinator has to come back: did all 30 segments finish? What if 2 crashed — and what if the coordinator itself dies and reruns?

The win here: fire all the encodes at once, then wait on them as a batch — collect what finished, isolate what failed, and make a coordinator retry safe so it never double-encodes. This is the other half of fan-out: L12 scattered the work, L13 picks the pieces back up.

1 · Fire all, then gather

timeout=0 returns the instant each run is created, so the loop dispatches all 30 in a blink. To learn their fates, wait on each afterwards — poll its state until final and read the finished FlowRun back: 

import time
from prefect import flow
from prefect.deployments import run_deployment
from prefect.client.orchestration import get_client

@flow   # deployed to the `extract` pool (a SYNC flow)
def process_archive(archive: str):
    segments = unpack(archive)

    runs = [                                    # ── fan out (L12): create all, don't wait
        run_deployment("encode-segment/encode",
                       parameters={"segment": s}, timeout=0)
        for s in segments
    ]
    finished = gather([r.id for r in runs])     # ── gather: now wait on the batch


def gather(run_ids):
    """Poll every run to a final state, then return them in order."""
    pending, done = list(run_ids), {}
    with get_client(sync_client=True) as client:
        while pending:
            time.sleep(2)
            pending = [rid for rid in pending
                       if not _settle(client.read_flow_run(rid), done)]
    return [done[rid] for rid in run_ids]

def _settle(fr, done):                          # final? stash it and drop from polling
    if fr.state.is_final():
        done[fr.id] = fr
    return fr.state.is_final()

Why poll, not wait_for_flow_run? The obvious helper, wait_for_flow_run, is async-only in our 3.7.5 build — calling it from this sync flow returns a coroutine ('coroutine' object has no attribute 'state'). So a sync coordinator polls state with get_client(sync_client=True). Make the flow async and you'd await wait_for_flow_run(r.id) instead — that's the next lesson, L14.

Why not just drop timeout=0? Letting run_deployment block (its default) waits on each encode serially — segment 2 can't start its wait until segment 1 finishes. You'd lose all the fleet parallelism L12 bought. Fire every run first, then wait — all 30 encode at once while the coordinator watches.

2 · Survive partial failure

Across 30 encodes on heterogeneous VMs, one will trip — a corrupt segment, an OOM, a flaky disk. By default reading a failed run's result re-raises the exception, which would sink the whole coordinator and lose the 29 good results. Pass raise_on_failure=False to get the outcome back as a value and decide for yourself: 

    done, failed = [], []
    for fr in finished:
        if fr.state.is_completed():
            done.append(fr.state.result())                 # the encoded segment
        else:
            failed.append(fr.parameters["segment"])        # don't raise — record it

    if failed:                                             # re-dispatch ONLY the failures
        retry_segments(failed)
    return {"ok": len(done), "failed": failed}
Gathering results vs states (ties back to L4) Each encode ran on a different machine. Its return value only comes back if the deployment persists results to shared storage (your MinIO bucket) — otherwise state.result() has nothing to deserialize. The run's state (Completed / Failed) always lives in the server DB, so checking whether each finished works regardless. For encodes, persist the output path, not the bytes.

3 · Make the retry safe — idempotency_key

Now the nasty case: the coordinator crashes after dispatching but before gathering, and Prefect retries it. Without protection it fans out all 30 encodes again — double the CPU, duplicate outputs. An idempotency_key built from stable inputs fixes this: a second create with a key Prefect has already seen returns the existing run instead of making a new one. 

        run_deployment("encode-segment/encode",
                       parameters={"segment": s},
                       timeout=0,
                       idempotency_key=f"{archive}:{s}")   # stable, NOT random
The key must be deterministic Derive it from the work itself (archive + segment), so the same segment always produces the same key. A random/UUID key defeats the whole point — every retry looks new. Same key → Prefect reuses the run; the already-completed encodes are skipped, not redone.
