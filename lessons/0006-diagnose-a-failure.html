Lesson 6 · Observability

Diagnose a failed run

It's 9am. A scheduled run failed at 3am while you slept. With a cron + print() setup you'd be squinting at log files. With Prefect, the run is sitting in history with its state, its logs, and the exact line that blew up — waiting for you.

This is the last of the four mission goals: read run history after the fact to work out why a past run failed. You've scheduled a flow (L5) — now you need to debug one without re-running it. This lesson's win: take a run that already failed and pin down which task broke, why, and when — entirely from history.

1 · Make a run fail (on purpose)

06_diagnose.py is in your repo: it fetches two rows, then a validate task chokes on a row with a missing amount. That's a permanent failure — bad input, not a transient blip (L2), so no retry would save it. Run it once to create the failed run: 

uv run python 06_diagnose.py     # exits with the flow in a Failed state

Now pretend you didn't see that terminal output — it ran at 3am. Everything below works purely from what Prefect recorded.

2 · Find the failure in the UI

Open localhost:4200 → Flow Runs, and filter by state Failed. There's nightly-report. Click it and you get the payoff cron never gave you:

  • A timeline of task runs — fetch_batch Completed, then validate Failed. You can see exactly how far the pipeline got.
  • The failed task's logs, including the traceback and the line ValueError: row 2 has no amount.
  • Timestamps, inputs, and the run's parameters — the full context of that one execution.

Recall the state vocabulary from the glossary: this is Failed (your code raised), not Crashed (the machine died). That tells you to look in your code, not your infrastructure.1

3 · Same diagnosis from the CLI

When you don't want to leave the terminal, the prefect flow-run commands read the same history.2 Start by listing just the failures: 

uv run prefect flow-run ls --state Failed --limit 5
┃  ID                              ┃ Flow         ┃ Name          ┃ State  ┃
│  d9975666-dff5-4271-…            │ nightly-rep… │ subtle-chinch… │ Failed │

Copy the ID, then inspect it for the state and the failure message, or pull the full logs: 

uv run prefect flow-run inspect d9975666-dff5-4271-b2db-5ceba26fbc8f
uv run prefect flow-run logs    d9975666-dff5-4271-b2db-5ceba26fbc8f
state=State(type=StateType.FAILED,
    message='Flow run encountered an exception: ValueError: row 2 has no amount')
...
ERROR | Flow run 'subtle-chinchilla' - Task run failed with exception:
       ValueError('row 2 has no amount')

Same smoking gun as the UI: validate failed on row 2. You diagnosed a 3am failure at 9am without re-running a thing.

4 · Once you know the fix

Found and fixed the bad data? You don't have to wait for the next schedule — replay the same run:2 

uv run prefect flow-run retry d9975666-dff5-4271-b2db-5ceba26fbc8f
The four diagnosis tools ls --state Failed (find it), inspect (state & message), logs (the traceback), retry (re-run once fixed). They're on the diagnosis cheatsheet.
