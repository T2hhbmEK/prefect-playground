Lesson 12 · Topology & fan-out

Fan out across the fleet

L9 spread tasks within one run — but on one machine. L11 capped each machine. Now the missing piece: one unpack job spawns dozens of encodes — how do you make the whole fleet chew through them at once, without light jobs starving behind heavy ones?

A real job is a pipeline: unpack one archive (light, I/O), then encode the dozens of segments inside it (heavy, CPU). The win here: run the unpack on its own pool, then dispatch each encode as an independent flow run so every machine works at once — and a flood of light jobs can never starve the encodes.

2 · Split the pools by job type

Two pools: extract (I/O — a generous --limit) and encode (CPU — GCL-capped, from L11). Each machine runs two workers, one per pool, each with its own limit. 

uv run prefect work-pool create extract --type process
uv run prefect work-pool create encode  --type process
Why not one shared pool? Head-of-line blocking. Share one pool and a backlog of slow, tightly-capped encodes parks every worker slot — your quick unpacks wait behind them (and vice versa). Separate pools give each job type independent limits, independent scaling, and clean routing. (One pool with two queues is the lighter alternative; two pools is cleaner and lets a future GPU box be its own pool.)

3 · Fan out — the crux

The trap: write the batch as one flow that .map()s the encodes. L9 says .map spreads tasks — true, but across threads/processes in one run, on one machine. Your other machines idle while one box grinds. "Subflow" means two different things:

You writeIt runs…
encode(seg) — a direct callinline, in the parent's process — one machine
run_deployment("encode/encode", …)a separate flow run on the encode pool — any worker, any machine
from prefect import flow
from prefect.deployments import run_deployment

@flow   # deployed to the `extract` pool
def process_archive(archive: str):
    segments = unpack(archive)              # light I/O — runs here
    runs = [
        run_deployment(
            "encode-segment/encode",        # "flow-name/deployment-name"
            parameters={"segment": s},
            timeout=0,                      # fire-and-forget: return at once
        )
        for s in segments
    ]
    return [r.id for r in runs]             # one distributed flow run per segment
Verified behaviour (3.7.5) timeout=0 returns the moment each run is created (don't wait); the default blocks until it finishes. And as_subflow=True (the default) only nests each encode under the parent in the UI — it does not pin execution to the parent's machine; the run still executes on the encode pool.1
