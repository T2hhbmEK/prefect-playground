Lesson 7 · Alerting

Get alerted when it breaks

Lesson 6 let you investigate a failure after you noticed it. But you shouldn't have to notice. The last mile of "visible when it breaks" is Prefect reaching out to you the moment a run fails — so you find out from a ping, not from a confused user.

You can diagnose a failed run (L6), but only once you go looking. This lesson's win: make a failing flow send you an alert automatically, and watch it arrive — fully locally. We'll use a throwaway local listener now; swapping in real Slack later is one line.

1 · A failure hook

Prefect flows take state-change hooks — functions it calls when the run enters a given state. on_failure is the one you want; it receives the flow, the flow run, and the failing state.1 

def alert_on_failure(flow, flow_run, state):
    msg = f"Flow '{flow_run.name}' entered {state.name}.\nReason: {state.message}"
    get_alerter().notify(body=msg, subject="🔴 Prefect: a run failed")

@flow(on_failure=[alert_on_failure])
def nightly_report():
    ...

The .notify() call belongs to a notification block — Prefect's reusable connection to a channel (Slack, a webhook, email).1 For learning, ours posts to a tiny local catcher.

2 · Watch an alert arrive

Two scripts are in your repo: alert_listener.py (a local webhook catcher) and 07_alerts.py (the failing flow + hook). In one terminal, start the listener: 

uv run python alert_listener.py     # waits on http://127.0.0.1:8077

In another, run the flow — it fails on purpose, like Lesson 6:

uv run python 07_alerts.py

The flow goes Failed, you see Running hook 'alert_on_failure' in its logs, and the listener terminal lights up: 

🔔 ALERT RECEIVED:
🔴 Prefect: a run failed
Flow 'translucent-silkworm' entered Failed.
Reason: Flow run encountered an exception: ValueError: row 2 has no amount

That's the win — nobody was watching the UI, and the failure still found you.

3 · Same hook, real Slack

In production you don't run a catcher — you point a real notification block at a Slack incoming webhook. The hook body is identical; only the block changes:1 

from prefect_slack import SlackWebhook

SlackWebhook(url="https://hooks.slack.com/services/...").save("ops-alerts")
# then in the hook:
SlackWebhook.load("ops-alerts").notify(body=msg, subject="...")

This is the one place the mission's "local for now" boundary meets the real world — and it's a single saved block away.

4 · The gotcha: hooks die with the process

Failed vs Crashed, again on_failure runs inside your flow's process. That's fine for Failed (your code raised, the process is alive to run the hook). But if the process is killed — OOM, a SIGKILL, the machine rebooting — it lands Crashed and there's no process left to send anything. The hook simply never runs.

That's what server-side Automations are for: rules the Prefect server evaluates on the events it receives — "when any flow run enters Failed or Crashed, send a notification."2 Because the server runs them, they catch crashes a dead process can't report, and one rule covers every deployment without editing a single flow. Set one up in the UI under Automations (trigger: flow run state → action: send a notification). Use hooks for per-flow logic; use Automations for fleet-wide safety nets.
