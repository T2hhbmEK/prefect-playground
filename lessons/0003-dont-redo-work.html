Lesson 3 · Caching

Don't redo work

Retries re-run work that failed. Caching does the opposite: it lets a task that already succeeded hand back its old result instantly — no slow API call, no re-download, no waiting. Re-run a pipeline at 9am and the parts that haven't changed just… skip.

In Lesson 2 you made a flaky task try again. But some work is slow and expensive even when it succeeds — a rate-limited API, a big download, a heavy computation. If the inputs haven't changed, running it twice is pure waste. This lesson's win: make an expensive task skip its own work the second time, and watch it land in a new state — Cached.

1 · One argument, again

Like retries, caching is a decorator argument. Add cache_policy= and Prefect computes a cache key for each call; if it has seen that key before, it returns the stored result instead of running your function.1 The simplest useful policy is INPUTS — key on the arguments: 

from datetime import timedelta
from prefect import task
from prefect.cache_policies import INPUTS

@task(cache_policy=INPUTS, cache_expiration=timedelta(minutes=5))
def expensive_fetch(source):
    time.sleep(3)                 # pretend this is a slow API
    return {"source": source, "rows": 42}

Call it twice with source="api://customers" and only the first call sleeps. The second returns instantly. cache_expiration is a timedelta — after it passes, the next call does the real work again (data goes stale; you decide how stale is too stale).1

2 · Run it and watch the Cached state

A caching example has been added to your repo. With the local stack from Lesson 1 up (docker compose up -d), run it: 

uv run python 03_caching.py

The flow calls expensive_fetch three times: same input twice, then a different input. In the logs (and the UI) you'll see: 

Task run 'expensive_fetch-564' - Finished in state Completed()
Task run 'expensive_fetch-b30' - Finished in state Cached(type=COMPLETED)
Task run 'expensive_fetch-969' - Finished in state Completed()

The middle call is Cached — same source, so Prefect skipped the 3-second sleep entirely. The third call used a different source, missed the cache, and did the work. Open the run in the UI and find the Cached task: that's a result you never recomputed.

3 · The gotcha: caching across runs

Here's the part that surprises people. Prefect's default cache policy is a composite:2 

DEFAULT = INPUTS + TASK_SOURCE + RUN_ID

That RUN_ID is the catch. It bakes the current flow run's id into the cache key — so the default cache only ever hits within a single run. Start a brand-new run tomorrow and every key is different; nothing is reused. That's safe, but it's not the win you came for.

Why this matters for the mission Your goal is pipelines that re-run on a schedule. To skip re-fetching data that hasn't changed between scheduled runs, you must drop RUN_ID — use a policy like INPUTS (or INPUTS + TASK_SOURCE). That's exactly why 03_caching.py uses INPUTS.

Proof: run the script a second time. Every task comes back Cached and the whole run finishes in under a second — the results persisted to disk from the first run.2

4 · Mixing policies

Policies combine with +. The most common upgrade over plain INPUTS is to also key on your code, so editing the task busts the cache:2 

from prefect.cache_policies import INPUTS, TASK_SOURCE

@task(cache_policy=INPUTS + TASK_SOURCE)   # change the inputs OR the code -> re-run
def transform(rows): ...

Other built-ins: FLOW_PARAMETERS (key on the parent flow's params), TASK_SOURCE (code only), and NO_CACHE to switch caching off entirely.2 They're all on the caching cheatsheet.
