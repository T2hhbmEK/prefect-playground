Lesson 15 · Putting it together

Run the fleet

Four lessons of theory — L11 per-worker GCL, L12 split pools + fan-out, L13 gather, L14 the async coordinator — and nothing actually run. Time to stop reading and execute the whole thing on your Docker stack.

The win here: one runnable script (10_fleet.py) and a handful of commands stand up the entire fan-out fleet — two pools, two workers, a per-machine GCL — and you watch the encodes gate themselves to two at a time even though the coordinator fired all six at once. This is L11–L14 made real, on one host standing in for the VM fleet.

0 · What you're running

10_fleet.py defines two flows and deploys both — no new concepts, just the L12/L13 pattern wired end to end with the L14 async coordinator:

DeploymentPoolDoes
process-archive/extractextractThe async coordinator: unpacks (light I/O), then fans out one arun_deployment per segment with asyncio.gather and gathers them.
encode-segment/encodeencodeThe CPU job: gates the (simulated) encode behind the per-worker GCL, persists its output path.

1 · Stand up the fleet

Each line is one earlier lesson, made concrete:

# two pools, split by job type (L12) — no head-of-line blocking
uv run prefect work-pool create extract --type process
uv run prefect work-pool create encode  --type process

# this box's CPU budget: limit = cores, occupy = threads/encode (L11)
# 4 ÷ 2  →  exactly 2 encodes run at once on this machine
uv run prefect gcl create encode:local-1 --limit 4

# register BOTH deployments (L8) — this only registers; it runs nothing
uv run python 10_fleet.py

# one worker per pool, EACH carrying its WORKER_ID so it finds its GCL (L11)
WORKER_ID=local-1 uv run prefect worker start --pool extract --limit 4 --name extract-1
WORKER_ID=local-1 uv run prefect worker start --pool encode  --limit 6 --name encode-1
Two workers, two terminals Start each worker start in its own shell (they run in the foreground and stay attached). The encode worker's --limit 6 lets it pick up all six runs; the GCL — not the worker limit — is what keeps only two encoding at once. The other four are live processes parked on concurrency() (the L11 "blocked run still holds a slot" gotcha, seen for real).

2 · Fire one archive

With both workers polling, trigger the coordinator:

uv run prefect deployment run process-archive/extract -p archive=clip01.tar

The encode worker's log shows the fan-out gated into three waves of two:

[local-1] encoding clip01.tar segment 4 (holding 2 slots)
[local-1] encoding clip01.tar segment 1 (holding 2 slots)   # wave 1: two at once
[local-1] done → encoded/clip01.tar/seg-4.ivf
[local-1] done → encoded/clip01.tar/seg-1.ivf
[local-1] encoding clip01.tar segment 2 (holding 2 slots)
[local-1] encoding clip01.tar segment 0 (holding 2 slots)   # wave 2 only after wave 1 frees slots
...
process-archive — 6 ok, 0 failed: []                        # coordinator gathered them
This is the whole arc in one run All six fired instantly (timeout=0, L12). The GCL admitted ⌊4 / 2⌋ = 2 (L11) — segments 3 & 5 wait until 4 & 1 finish. The async coordinator awaited them all (asyncio.gather + wait_for_flow_run, L14) and returned the summary. Bump the GCL to --limit 8 (gcl update) and rerun: four at a time. Same flow code — only the number changed.

3 · When one segment dies

Restart the encode worker with FAIL_SEGMENT=3 and fire a fresh archive. Segment 3 raises; the coordinator does not crash: 

WORKER_ID=local-1 FAIL_SEGMENT=3 uv run prefect worker start --pool encode --limit 6
uv run prefect deployment run process-archive/extract -p archive=clip02.tar
# →  process-archive — 5 ok, 1 failed: [3]

That's the L13 partial-failure pattern (await fr.state.aresult(raise_on_failure=False) in the async coordinator): read the bad run's outcome as a value instead of re-raising, keep the five good results, record the one failure to re-dispatch.

4 · Rerun is free

Trigger the same archive again. The idempotency_key=f"{archive}:{s}" means every segment's key is already known, so Prefect returns the existing runs — zero new encodes, the worker stays idle: 

uv run prefect deployment run process-archive/extract -p archive=clip01.tar
# →  6 ok, 0 failed: []     (encode worker logs NOTHING — all runs reused)

Why this matters A coordinator that crashes after dispatching can safely retry: it picks up the finished encodes instead of doubling the CPU. A random/UUID key would defeat this — every retry would look new and re-encode everything (L13).

5 · It gathered the async way (L14)

Why 6 ok came back cleanly The process-archive coordinator is an async def flow (L14): it fired every encode with arun_deployment(timeout=0), then gathered with await asyncio.gather(*(wait_for_flow_run(id) …)) — the documented API, which only works under await in this build. One subtlety it bakes in: wait_for_flow_run is event-driven and its returned state can race the commit (a finished run can momentarily read back RUNNING, ~1 run in 5), so the coordinator re-reads each run's authoritative state before classifying. Without that re-read, the very run you just did would sometimes report a spurious 1 failed — exactly what "actually run it" surfaces. Full story in L14.
