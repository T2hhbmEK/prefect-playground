Lesson 8 · Scaling out

Work pools & workers

serve() was one process that did everything — and died when you closed the terminal. The grown-up model splits that in two: deployments describe what to run, and workers decide where to run it. This is the door to scaling out and, eventually, production.

Two lessons flagged the same ceiling: serve() is the scheduler (L5), and an in-process hook can't survive a dead process (L7). Both point at the same fix. This lesson's win: deploy a flow to a work pool, start a worker, and watch the worker — not you — execute the run. This is the first step of the expanded mission.

1 · Three pieces, cleanly separated

PieceRole
DeploymentThe what: your flow + schedule + parameters, pointed at a pool.
Work poolThe queue: a typed bucket that scheduled runs land in, waiting to be picked up.
WorkerThe where: a process that polls a pool and runs whatever it finds.

Runs flow one way: deployment → pool → worker.1 With serve(), one process was all three at once. Splitting them means the schedule lives on the server (it survives any terminal), and you can point many deployments at one pool, or run several workers for more throughput.

2 · Create a pool, deploy to it

A process work pool runs each flow run as a local subprocess — no Docker, no Kubernetes, perfectly local:1 

uv run prefect work-pool create local-process --type process

Then deploy. Because a worker runs in a separate process, it needs to know where your code lives — from_source() tells it:2 

worker_pipeline.from_source(
    source=str(Path(__file__).parent),          # load code from this folder
    entrypoint="08_workers.py:worker_pipeline",
).deploy(
    name="worker-pipeline",
    work_pool_name="local-process",
)

08_workers.py is in your repo. Run it once — it registers the deployment and exits immediately: 

uv run python 08_workers.py
deploy() ≠ run Unlike serve(), deploy() doesn't run anything — it just records the deployment. If you trigger a run now, it sits Scheduled forever. Nothing executes until a worker is polling the pool. This trips up everyone once.

3 · Start a worker and watch it work

In one terminal, start a worker on that pool — leave it running:1

uv run prefect worker start --pool local-process

In another, trigger a run of the deployment:

uv run prefect deployment run 'worker-pipeline/worker-pipeline'

Watch the worker's terminal — it picks the run up and executes it:

Flow run 'furry-pigeon' - Running 1 deployment pull step(s)
Flow run 'furry-pigeon' - Beginning flow run for flow 'worker-pipeline'
Task run 'fetch_rows-69e' - Finished in state Completed()
Flow run 'furry-pigeon' - [from-a-worker] processed 42 rows — executed by a worker, not by me
Flow run 'furry-pigeon' - Finished in state Completed()

The run executed in a process you didn't launch by hand and that isn't tied to your trigger command. Add a cron= to the deploy and the server schedules the runs; the worker executes them — around the clock, with no serve() process babysitting anything.

4 · serve() or workers?

serve()work pool + worker
ProcessesOne (does everything)Server schedules · worker(s) execute
Survives a closed terminalNoYes (worker is its own service)
Scale / many deploymentsAwkwardNatural — more workers, shared pools
Path to remote infraNoneSwap pool type (Docker, K8s…) later
Best forDev, a single flowAnything heading toward production

Same deployment concept either way — you've just moved the execution out into a worker. Details on the work pools cheatsheet.
