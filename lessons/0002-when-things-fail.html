Lesson 2 · Resilience

When things fail

Networks blip. APIs return 503 for two seconds and then recover. A cron job would die and page you at 3am. A resilient flow just tries again — and you find out in the morning it healed itself.

In Lesson 1 you learned to watch a run and read its states — including Failed, the state a task lands in when your code raises an error.1 But many failures aren't really failures: they're a momentary blip that would succeed on a second attempt. This lesson's win: make a flaky task retry itself, and watch it recover live.

1 · One argument buys you resilience

Add retries= to a task and Prefect re-runs it automatically when it raises. Add retry_delay_seconds= to wait between attempts.2 That's the whole core idea: 

@task(retries=3, retry_delay_seconds=2)
def fetch_flaky_data(source):
    if random.random() < 0.6:        # fails ~60% of the time
        raise ConnectionError("transient blip")
    return {"rows": 42}

retries=3 means up to 4 total attempts (the first try plus three retries). With a 60% failure rate, the odds that all four attempts fail are only ~13% — so most of the time this flaky task recovers on its own.

2 · Run it and watch the retries

Make sure your local stack from Lesson 1 is up (docker compose up -d). Then run the resilience example that's been added to your repo: 

uv run python 02_resilience.py

Open the run in the UI and click the fetch_flaky_data task. This is the payoff: you'll see multiple attempts — some Failed with the ConnectionError in the logs — and then a final Completed. The flow stays green even though individual attempts failed. That row of attempts is something cron could never show you.

Try it — make it lose Run the script a handful of times. Eventually you'll hit the ~13% case where all four attempts fail and the flow goes Failed. Compare that run's task to a successful one: same code, different luck. This is why you set retries higher for flakier dependencies.

3 · Smarter waiting: backoff

Retrying every 2 seconds can keep a struggling service down. Better to back off — wait longer each time. Pass a list, or use the built-in helper:2 

from prefect.tasks import exponential_backoff

@task(retries=4, retry_delay_seconds=exponential_backoff(backoff_factor=2))
def fetch_with_backoff(source):   # waits 2s, 4s, 8s, 16s
    ...

(This task is in 02_resilience.py too — swap the flow to call it if you want to watch the growing delays in the timeline.)

4 · Don't retry what can't recover

Retries are for transient failures. A 404 or a bad password (401) won't fix itself — retrying just wastes time. Pass retry_condition_fn to decide per-failure whether to bother:2 

def retry_handler(task, task_run, state) -> bool:
    try:
        state.result()
    except httpx.HTTPStatusError as exc:
        return exc.response.status_code not in (401, 404)  # give up on these
    return True

@task(retries=3, retry_condition_fn=retry_handler)
def api_call(url): ...

5 · Cap runaway runs: timeout

A retry can't help a task that hangs forever. timeout_seconds cancels a run that overstays its welcome:3 

@flow(timeout_seconds=30)
def pipeline(): ...   # killed if it runs longer than 30s
