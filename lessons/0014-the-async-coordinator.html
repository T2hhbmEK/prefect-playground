Lesson 14 · Scaling out

The async coordinator

L13 gathered a fan-out with wait_for_flow_run — but in our build that function is async-only. So a coordinator that gathers the documented way has to be an async def flow. Here's how to write it — before you stand the whole fleet up and run it in L15.

The win: write the fan-out coordinator as one async def flow — arun_deployment to fire, await wait_for_flow_run to gather, asyncio.gather to run both phases concurrently — no hand-rolled poll loop. This is the coordinator in 10_fleet.py; L15 runs it end-to-end.

0 · Why the coordinator is async

L13 introduced wait_for_flow_run(r.id) as the way to gather a fan-out. The catch: in 3.7.5.dev4 it's a coroutine — call it from a sync flow and you get 'coroutine' object has no attribute 'state'. You can hand-roll a poll loop with the sync client (the L13 cheatsheet shows it), but the moment you want the documented gather API, the coordinator must be async.

Verified in this build inspect.iscoroutinefunction(wait_for_flow_run) → True (no .aio). run_deployment is sync with an async .aio, and arun_deployment is importable from prefect.deployments as a true coroutine. State grows an aresult() coroutine alongside the sync result().

1 · The diff: sync → async

Same shape as L13's gather — fire all with timeout=0, then gather — but every blocking call becomes an awaited one, and the poll loop disappears:

Two phases, both concurrent asyncio.gather appears twice: once over the creates (phase 1) and once over the waits (phase 2). Each create fires with timeout=0 so it returns the instant the run exists — fire all, then wait, exactly as in L13.

2 · What async actually buys you (be honest)

Tempting to say "now the encodes run in parallel." They already do — timeout=0 (L12) fans the encodes out across the fleet and the GCL (L11) gates them, whether the coordinator is sync or async. Async changes none of that. What it changes:

Async gives youWhy it matters
The documented gather APIwait_for_flow_run only works under await here — async is the only way to use it instead of a hand-rolled loop.
Concurrent dispatchThe sync version issues N create-calls back-to-back. gather fires them at once — invisible at 6 segments, the whole cost at 600.
Less codeThe 15-line poll loop is deleted.
Cheap waitingOne coroutine per run, not a thread blocking — scales to thousands of in-flight encodes.

The honest takeaway: for the six-segment demo, async barely moves the wall clock — the encode time dominates. It earns its keep on dispatch at high fan-out and on using the intended API. Drag the slider:

3 · The catch this gather hides

A completed encode can come back marked "failed" — about once in five runs Run this coordinator on the fleet and you can hit a surprise: a live run reports 5 ok, 1 failed: [3] — but segment 3's encode had clearly completed (the worker logged done → …/seg-3.ivf, and the run was Completed on the server). The bug was in the gather. Reading the source explains it: wait_for_flow_run here is event-driven — its poll_interval is deprecated/ignored. It returns on the first final-state event, then re-reads the run — and that re-read can race the state commit, handing back a still-RUNNING FlowRun for a run that actually finished. Classify on it and a good encode looks failed. 
# robust: re-read the authoritative state before classifying (one read settles it)
async with get_client() as client:
    finished = await asyncio.gather(*(client.read_flow_run(fr.id) for fr in finished))
With the re-read, 8 of 8 reruns came back clean. This is the kind of API reality you only surface by running it for real — which you'll do in L15. The docstring hides it.

4 · Async flow mechanics

  • You don't call asyncio.run. Prefect runs your async def flow inside its own event loop — both when a worker executes it and when you call it directly.
  • Mixing sync and async: an async flow can only await coroutines. The unpack task is async too, so segments = await unpack(...) reads cleanly without bridging a loop.
  • Results: await fr.state.aresult(raise_on_failure=False). The fr.state.result.aio(...) form looks right but drops self in this build — use aresult.
