Lesson 15 · Scaling out

The async coordinator

L14 ran the fleet for real and hit a wall: wait_for_flow_run is async-only in our build, so the sync coordinator had to hand-roll a poll loop. Make the coordinator async and that loop disappears — and running it surfaces one more catch the slide never would.

The win: rewrite L14's coordinator as one async def flow — arun_deployment to fire, await wait_for_flow_run to gather, asyncio.gather to run both phases concurrently — deleting the hand-rolled poll loop entirely. The encode side is untouched; only the coordinator changes. New script: 11_fleet_async.py.

0 · The loose end from L14

L13 taught wait_for_flow_run(r.id) as the way to gather a fan-out. L14 ran it and found that in 3.7.5.dev4 the function is a coroutine — calling it from a sync flow gives 'coroutine' object has no attribute 'state'. The sync coordinator in 10_fleet.py worked around it by polling run state with the sync client in a hand-written loop. That loop is the smell this lesson removes.

Verified in this build inspect.iscoroutinefunction(wait_for_flow_run) → True (no .aio). run_deployment is sync with an async .aio, and arun_deployment is importable from prefect.deployments as a true coroutine. State grows an aresult() coroutine alongside the sync result().

1 · The diff: sync → async

Same shape as L14 — fire all with timeout=0, then gather — but every blocking call becomes an awaited one, and the poll loop is gone:

# 10_fleet.py (sync)                        # 11_fleet_async.py (async)
runs = [run_deployment(...) for s in segs]   runs = await asyncio.gather(*(
                                                 arun_deployment(..., timeout=0)
                                                 for s in segs))
finished = gather([r.id for r in runs])      finished = await asyncio.gather(*(
  # ^ hand-rolled poll loop over the            wait_for_flow_run(r.id)
  #   sync client (15 lines)                     for r in runs))
res = fr.state.result(...)                    res = await fr.state.aresult(...)

Two phases, both concurrent asyncio.gather appears twice: once over the creates (phase 1) and once over the waits (phase 2). Each create fires with timeout=0 so it returns the instant the run exists — fire all, then wait, exactly as in L13.

2 · What async actually buys you (be honest)

Tempting to say "now the encodes run in parallel." They already did — in L14, on one host. timeout=0 is what fans the encodes out across the fleet; the GCL is what gates them. Async changes none of that. What it changes:

Async gives youWhy it matters
The documented gather APIwait_for_flow_run only works under await here — async is the only way to use it instead of a hand-rolled loop.
Concurrent dispatchThe sync version issues N create-calls back-to-back. gather fires them at once — invisible at 6 segments, the whole cost at 600.
Less codeThe 15-line poll loop is deleted.
Cheap waitingOne coroutine per run, not a thread blocking — scales to thousands of in-flight encodes.

The honest takeaway: for the six-segment demo, async barely moves the wall clock — the encode time dominates. It earns its keep on dispatch at high fan-out and on using the intended API. Drag the slider:

3 · Run it

The pools, the GCL, and the encode-segment/encode deployment are all from L14 — unchanged. You only register the new coordinator:

# the async coordinator deploys ALONGSIDE the sync one (distinct flow name)
uv run python 11_fleet_async.py

# same two workers as L14, one per pool (encode worker carries its WORKER_ID → GCL)
WORKER_ID=local-1 uv run prefect worker start --pool encode  --limit 6 --name encode-1
uv run prefect worker start --pool extract --limit 4 --name extract-1

# fire it — note the deployment name ends in -async
uv run prefect deployment run process-archive-async/extract -p archive=clip01.tar
# →  process-archive-async — 6 ok, 0 failed: []   (waves of 2, same as L14)

Verified end-to-end on 3.7.5.dev4: 6 ok, 0 failed, the encode worker logging the same three waves of two — the GCL gates identically whether the coordinator is sync or async.

4 · The catch running it surfaced

A completed encode came back marked "failed" — once in five runs The first live run reported 5 ok, 1 failed: [3] — but segment 3's encode had clearly completed (the worker logged done → …/seg-3.ivf, and the run was Completed on the server). The bug was in the gather. Reading the source explains it: wait_for_flow_run here is event-driven — its poll_interval is deprecated/ignored. It returns on the first final-state event, then re-reads the run — and that re-read can race the state commit, handing back a still-RUNNING FlowRun for a run that actually finished. Classify on it and a good encode looks failed. 
# robust: re-read the authoritative state before classifying (one read settles it)
async with get_client() as client:
    finished = await asyncio.gather(*(client.read_flow_run(fr.id) for fr in finished))
With the re-read, 8 of 8 reruns came back clean. This is L14's lesson again: running it shows you the API reality the docstring hides.

5 · Async flow mechanics

  • You don't call asyncio.run. Prefect runs your async def flow inside its own event loop — both when a worker executes it and when you call it directly.
  • Mixing sync and async: an async flow can only await coroutines. The unpack task is async too, so segments = await unpack(...) reads cleanly without bridging a loop.
  • Results: await fr.state.aresult(raise_on_failure=False). The fr.state.result.aio(...) form looks right but drops self in this build — use aresult.
