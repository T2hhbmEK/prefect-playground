Lesson 9 · Parallelism

Distribute the work

Lesson 8 spread flow runs across workers. This is the other axis: spreading the tasks inside one run so they don't crawl through one at a time. One decorator argument turns a sequence of waits into an overlap — and, for heavy compute, onto multiple cores.

Work pools answer "where does the run execute?" (L8). Task runners answer a different question: "how do the tasks within one run spread out?" This lesson's win: take a batch of mapped tasks and watch them run together instead of one-by-one — and learn which runner fits which work.

1 · The runner only acts on submitted tasks

A task runner governs tasks you .submit() or .map(). Plain calls stay strictly sequential no matter which runner you pick — that's the first thing people miss. 

from prefect.task_runners import ThreadPoolTaskRunner

@flow(task_runner=ThreadPoolTaskRunner(max_workers=4))
def fetch_all():
    return fetch.map(range(4)).result()   # .map → these run together
    # fetch(0); fetch(1); ...             # plain calls → still one at a time

The default runner is already ThreadPoolTaskRunner — so your .map() in Lesson 1 has been running tasks concurrently all along.

2 · Threads vs processes — run it

09_task_runners.py is in your repo. It times the same batch of four tasks under two runners, for two kinds of work: 

uv run python 09_task_runners.py
4x fetch  (I/O) ThreadPool : 1.1s  # ~2s sequential
4x crunch (CPU) ThreadPool : 4.2s  # GIL: no speedup
4x crunch (CPU) ProcessPool: 2.4s  # real parallelism

Two lessons in three numbers:

  • I/O-bound work loves threads. Four half-second waits overlap into ~1s instead of ~2s. While one task waits on the network, the others proceed.
  • CPU-bound work doesn't. Four heavy loops on threads take longer than one (4.2s) — Python's GIL lets only one thread actually compute at a time. Switch to ProcessPoolTaskRunner and they run on separate cores: 2.4s.

3 · Why threads can't help CPU work

The GIL, in one line CPython runs only one thread of Python bytecode at a time (the Global Interpreter Lock). Threads still win for I/O because a waiting task releases the GIL — but a task that's computing holds it, so threads take turns. Only separate processes have their own interpreter, and their own core.

4 · …but processes aren't free

Don't reach for ProcessPoolTaskRunner by reflex. Each process must be spawned and each task's inputs/outputs pickled across the boundary — real overhead. For light CPU tasks that overhead can make processes slower than threads. Processes only pay off when each task does enough heavy computation to dwarf the startup cost. Measure, like the script does.
