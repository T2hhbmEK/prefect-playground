Part 1 · Recall from memory

Think the answer through completely, then reveal to check.

You write @task(retries=3). It fails every time. How many times does it run in total? L2
4 total — the first attempt plus 3 retries. retries=N means N+1 runs.
A run's state is Crashed, not Failed. Where do you look — and why? L1·L6
At the infrastructure (machine/process — OOM, killed), not your code. Failed = your code raised (there's a traceback); Crashed = the process vanished before it could report one.
Your cache_policy=INPUTS cache works within a run but never reuses results across scheduled runs when you switch to the default. What's the cause? L3
The default policy is INPUTS + TASK_SOURCE + RUN_ID. RUN_ID makes every run's keys unique, so it only hits within one run. Drop it (use INPUTS) for cross-run reuse.
Your task transcodes a 200 MB video. What should it return, and where does the mp4 itself go? L4
Return the object key (a small string); upload the mp4 to object storage yourself. Cache the pointer, store the payload. result_storage holds the return value, not the big file.
In one sentence: what two things does flow.serve() do? L5
It registers a deployment and runs a long-lived process that creates & executes its scheduled runs. Close the terminal → no more runs.
You ran deploy() and triggered a run, but it sits Scheduled forever. Why? L8
No worker is polling that work pool. deploy() only registers; a worker (prefect worker start --pool …) is what actually executes.
Why does ThreadPoolTaskRunner speed up I/O-bound tasks but not CPU-bound ones? L9
The GIL: only one thread runs Python bytecode at a time. A waiting task releases it (so I/O overlaps); a computing task holds it (so threads take turns). CPU parallelism needs ProcessPoolTaskRunner.
An overnight scheduled run failed. Name the CLI chain to diagnose it from history. L6
flow-run ls --state Failed → flow-run inspect <id> → flow-run logs <id> → flow-run retry <id> once fixed.
An on_failure hook sends your alerts. The process gets OOM-killed (Crashed). Does the alert fire? What's the fix? L7
No — the hook runs in the now-dead process. Use a server-side Automation (trigger on Failed/Crashed → notify); the server runs it independently, and one rule covers every deployment.
When does setting @flow(task_runner=…) do nothing? L9
When you call tasks the plain way. Runners only act on .submit() / .map()ed tasks — everything else stays sequential.
You moved an mp4 pipeline from local MinIO to Tencent COS. How much of the task code changes? L4
None. Both are S3-compatible — you swap only the S3Bucket block (MinIOCredentials + localhost → AwsCredentials + the COS endpoint). The upload_from_path / result_storage code is identical.
Two distribution axes: what does a work pool spread, vs a task runner? L8·L9
Work pool + worker spreads whole flow runs across infrastructure (set on the deployment). Task runner spreads the tasks within one run across cores/threads (set in flow code). Orthogonal — they compose.
