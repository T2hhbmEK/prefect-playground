Lesson 1 · Observability

See your flow run

Right now your flow runs and vanishes. The first step to a production pipeline is being able to watch it — and read what happened after it's gone.

You already have a working flow in 01_getting_started.py: it fetches customer IDs and maps a task over them. When you run it, you see some print-style output and then nothing. If it failed at 3am, you'd have no idea why.

Prefect's first promise isn't scheduling or retries — it's observability: every run is recorded as states, logs, and timings you can inspect.1 This lesson gets you one tangible win: run your existing flow and watch it appear in the UI.

1 · Start the local stack

The UI and the database that records runs come from a local server.2 This repo runs it (plus a Postgres database and MinIO object store) with Docker Compose, so one command brings up everything: 

docker compose up -d --build

Leave it running. Open http://127.0.0.1:4200 in your browser. You'll see an empty dashboard. That's expected: nothing has run against this server yet.

Note The server runs the official prefecthq/prefect image pinned (in docker-compose.yml) to the same Prefect version as your client, so the API versions match exactly. It stores every run in Postgres — that's what the UI reads. Stop the stack with docker compose down (your data survives in volumes); docker compose down -v wipes it.

2 · Point your flow at the server

The stack runs in the background, so you just tell Prefect where the API lives (once), then run the flow you already have: 

uv run prefect config set PREFECT_API_URL=http://127.0.0.1:4200/api
uv run python 01_getting_started.py

Now refresh the dashboard. Your run is there — with a randomly-generated name (like "vivacious-vole"), a state, and a duration.

3 · Read the run

Click the flow run. This is the payoff. You should see:

  • A top-level Completed state for the flow run.
  • A row of task runs — one get_customer_ids, then ten process_customer runs from your .map(), each with its own state.
  • Logs and a timeline showing what ran when.

That row of task states is the thing cron never gave you: when something breaks, you'll see exactly which task failed, not just "the script errored."

Try it — break something on purpose Add raise ValueError("boom") as the first line of process_customer, run again, and refresh. The flow run now shows Failed, and the task runs show you it was the mapped task — not the fetch — that died. Then remove the line. This is debugging a pipeline instead of guessing.

4 · The states vocabulary

Everything in that UI is built on states. You'll read these words constantly, so anchor them now:

State One-liner
Completed Finished successfully.
Running Executing right now.
Failed Your code raised an error it didn't handle.
Crashed The infrastructure died — Prefect lost contact.
Scheduled Will run later (you'll meet this when we add schedules).

Full list and definitions live in the glossary.
