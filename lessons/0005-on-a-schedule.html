Lesson 5 · Deployments & schedules

Put it on a schedule

So far you've run flows by typing python script.py. The whole point of the mission is to stop doing that — to hand the script to Prefect and have it run on its own, every night, whether or not you're watching. That's a deployment.

You've made flows observable (L1), resilient (L2), and cheap to re-run (L3, L4). They still only run when you launch them by hand. This lesson's win: turn a script into a deployment that runs itself on a schedule, and watch new runs appear in the UI with no one at the keyboard.

1 · serve() turns a script into a deployment

A deployment is your flow plus the how/when to run it — a schedule, default parameters, a name you can trigger by.1 The simplest way to create one locally is flow.serve():2 

@flow(log_prints=True)
def scheduled_pipeline(label="nightly"):
    n = fetch_rows()
    print(f"[{label}] processed {n} rows")
    return n

if __name__ == "__main__":
    scheduled_pipeline.serve(name="scheduled-pipeline", interval=30)

serve() does two things at once: it registers the deployment with the API, and it starts a long-running process that creates scheduled runs and executes them. No work pools, no workers, no containers — just this one process.2

2 · Run it and watch runs appear by themselves

05_deployment.py is in your repo. With your local stack up (docker compose up -d), start serving: 

uv run python 05_deployment.py     # leave it running; Ctrl-C to stop

It prints a deployment URL and then sits there. Every 30 seconds a fresh run fires on its own — you'll see lines like: 

Flow run 'diligent-duck' - View at http://127.0.0.1:4200/runs/flow-run/...
Task run 'fetch_rows-5c3' - Finished in state Completed()
Flow run 'diligent-duck' - [nightly] processed 42 rows

Open Deployments in the UI: there's scheduled-pipeline, with its schedule and a growing list of runs you didn't launch. That's the mission's headline, achieved.

Try it — trigger one yourself While serve() is running, open another terminal and run uv run prefect deployment run 'scheduled-pipeline/scheduled-pipeline'. An on-demand run appears alongside the scheduled ones — same deployment, no editing code. This is how you'd kick off a backfill.

3 · cron vs interval

Two ways to express "when". interval= is "every N seconds"; cron= is a standard cron string for clock-based times.1 

scheduled_pipeline.serve(name="nightly", cron="0 8 * * *")     # every day 08:00
scheduled_pipeline.serve(name="hourly",  interval=3600)        # every hour

The demo uses interval=30 so you don't wait until 8am to see it work; the comment in the script shows the realistic cron="0 8 * * *". Both create the same kind of Scheduled runs you met in Lesson 1 — Prefect just makes them for you now.

4 · The gotcha: serve() must keep running

No process, no runs serve() is the thing executing your schedule. Close the terminal and the schedule goes dark — no runs fire. For real use you'd keep it alive with a process manager (launchd, tmux, a systemd unit). The heavier alternative — flow.deploy() with work pools and workers — survives a logout but needs infrastructure we're keeping out of scope for now.
