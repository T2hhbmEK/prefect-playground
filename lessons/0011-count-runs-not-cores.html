Lesson 11 · Concurrency limits

Count runs, not cores

In L9 you reached for ProcessPoolTaskRunner to push CPU work onto many cores. This is the opposite problem: when the work is already maximally parallel — a CPU-pinning AV1 encode in an external process — how do you stop a fleet of mismatched machines from running too much at once?

Your encode flow barely computes in Python — it shells out: subprocess.run(["ffmpeg", …]) driving SVT-AV1. The win here: size a per-machine concurrency cap so heterogeneous VMs stay busy without ever oversubscribing CPU — even though Prefect can't see a single one of the encoder's threads.

1 · The work is invisible to Prefect

The heavy CPU work, and the 3–4 threads it spreads across, live inside the child process. To Prefect the task is just one box in Running — it has no idea four cores are pinned. So the L9 reflex — "CPU-bound? reach for ProcessPoolTaskRunner" — buys you nothing here: there's no Python loop to spread across cores; the parallelism is already inside ffmpeg. Threads-vs-processes is the wrong question. The right one is how many encodes do I let run at once on this machine?

Pin the encoder first — or nothing else matters ffmpeg, SVT-AV1, and aomenc auto-detect every core and grab them by default. Before any orchestration, fix the thread count: -threads 4 (ffmpeg), --lp 4 / -svtav1-params lp=4 (SVT-AV1), --threads=4 (aomenc). Now each encode has a known cost. Skip this and no Prefect limit can save you — one un-pinned encode swallows the whole box.

2 · Every Prefect limit counts runs

This is the mental shift. Prefect's concurrency controls count runs — never cores, never the threads inside your subprocess. It can't see them, so it will never protect your CPU automatically. You do the arithmetic — cores ÷ threads-per-encode — and hand Prefect the number.

3 · Three limits — only one is per-machine

LimitScopeGood for
Work pool / queue limit
work-pool set-concurrency-limit		 Fleet-global — shared by every worker polling the pool A global cap: a shared NAS, licence seats, total egress
Worker run limit
worker start --limit N		 Per worker, but counts whole flow runs A blunt per-machine cap; serialises the entire flow
Global Concurrency Limit + occupy, named per worker Per machine, weighted Exactly this — cap CPU saturation per box

You met the first one on the work-pools cheatsheet — and it's the trap. Set a pool limit to 4 and your whole fleet runs 4, not 4-per-box. The worker --limit is per-machine but blunt: a light unpack job waits behind a heavy encode. The precise tool is a Global Concurrency Limit (GCL) you acquire in code and name per worker.

4 · The per-machine recipe

# 1. One limit per machine, sized to its cores (create it first —
#    concurrency() will NOT auto-create it). The 16-core box:
uv run prefect gcl create encode:big-cvm-1 --limit 16

# 2. Start the worker carrying its own identity
WORKER_ID=big-cvm-1 uv run prefect worker start --pool encode --limit 8
# 3. Wrap the subprocess, occupying its thread cost
import os, subprocess
from prefect.concurrency.sync import concurrency

worker_id = os.environ["WORKER_ID"]

with concurrency(f"encode:{worker_id}", occupy=4):   # 4 threads → 4 slots
    subprocess.run(["ffmpeg", "-threads", "4", ...], check=True)
The whole idea in one line The limit is the machine's core budget; occupy is each encode's cost. The 16-core box (--limit 16) admits four 4-thread encodes; the fifth blocks until one frees. The 4-core VM runs the same flow code — just gcl create encode:small-1 --limit 4 → one encode. Heterogeneous fleet, one flow, a different budget per box.
Why this beats --limit 1 The GCL gates only the encode block, so the flow's light unpack/setup tasks still overlap — "Using --limit 1 on the worker forces entire flow runs to be sequential. But often only specific tasks need limits." The server coordinates the count across every worker's separate processes.1
