Part 1 · Recall from memory

Think the answer through completely, then reveal to check. Tags are shuffled so you can't coast on context.

Your coordinator fires each encode with run_deployment and waits on it before creating the next. What did you just break? L13
The fleet parallelism. Create every run with timeout=0 first, then gather them. Blocking as you create serialises the fan-out — the encodes run one at a time across the whole fleet.
An encode shells out to ffmpeg with 4 threads. You reach for ProcessPoolTaskRunner to cap CPU. Why does it do nothing? L11
The heavy work and its threads live in a child process Prefect can't see — there's no Python compute to parallelise, so the task runner is the wrong lever. Cap it with a concurrency limit instead. Count runs, not cores.
You rewrite the sync coordinator as async. Do the encodes now finish faster? L14
No. The encodes already ran in parallel — that's timeout=0 + the GCL, in both versions. Async buys the documented (async-only) wait_for_flow_run API, concurrent dispatch (matters at 600 segments, invisible at 6), and deleting the poll loop — not wall-clock on small batches.
Inside a coordinator: encode(seg) vs run_deployment("encode/encode", timeout=0). What's the execution difference? L12
The direct call runs inline on the coordinator's machine. run_deployment creates a separate flow run on the encode pool → picked up by any worker, any box. .map also stays on one machine — only a deployment escapes the host.
A 16-core box, each encode pinned to 4 threads. What's the GCL limit, the occupy, and how many run at once? L11
limit = 16 (the core budget), occupy = 4 (each encode's thread cost) → ⌊16/4⌋ = 4 concurrent. The same flow code handles a 32-core box by changing only the limit number.
Six encodes all fire at once. The worker was started with --limit 6; the GCL is limit 4 / occupy 2. How many encode simultaneously? L15
Two (⌊4/2⌋) — in three waves of two. The worker --limit lets all six be picked up, but the GCL, not the worker limit, caps actual concurrency. (The picked-up-but-parked processes are the real gotcha.)
Why must idempotency_key come from stable inputs like f"{archive}:{seg}" — never a UUID? L13
A re-created run with a seen key returns the existing run, so a crashed-and-rerun coordinator reuses finished encodes instead of doubling CPU. A UUID is new every time → every rerun looks new → it re-encodes everything. Random keys defeat the whole point.
Three scopes can cap concurrency. Which one is the trap for per-machine saturation, and why? L11·L12
The pool/queue limit (set-concurrency-limit) — it's fleet-global, shared by every worker, so it can't saturate one box. For per-machine you need a GCL named per worker (encode:local-1) + occupy. (The worker --limit is per-worker but blunt — it counts whole flow runs.)
wait_for_flow_run hands back a Running FlowRun for a run the worker already logged as done (~1 in 5). Why — and the fix? L14
It's event-driven (poll_interval is ignored): it returns on the first final-state event, then re-reads the run — and that re-read can race the state commit. Fix: re-read each run's authoritative state (read_flow_run) before classifying completed vs failed. (Also: use aresult, not state.result.aio.)
Does as_subflow=True on run_deployment make the dispatched run execute on the caller's machine? L12
No — it's lineage/UI only (the parent–child link in the run graph). The run still executes on its pool, wherever a worker picks it up. Co-location is about where the work runs, not the lineage flag.
An encode ran on another machine and returned an output path. Why might the coordinator get None back instead? L13·L4
The deployment didn't persist its result to shared storage (MinIO/S3) — return values cross machines only via persisted storage. The run state always lives in the server DB, so checking whether it finished works regardless. Persist the path, not the bytes.
You write with concurrency("encode:local-1", occupy=4): and it errors that the limit doesn't exist. What's missing? L11
Create the limit first: prefect gcl create encode:local-1 --limit 16. concurrency() does not auto-create a GCL — it expects one already there.
