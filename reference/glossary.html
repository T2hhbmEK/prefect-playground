Prefect Glossary

The vocabulary, defined once and used consistently across every lesson.

Core objects

TermMeaning
FlowA Python function decorated with @flow. The unit Prefect orchestrates: it gets observed, can be deployed, scheduled, and retried. One flow run = one execution.
TaskA Python function decorated with @task, called from inside a flow. The smallest unit Prefect tracks. Tasks can retry, cache, and run concurrently independently of each other.
SubflowA flow called from inside another flow. Tracked as its own run, nested under the parent.
RunA single execution of a flow or task. Has an ID, a state, logs, timings, and (for tasks) a result.
StateWhere a run is in its lifecycle (see below). Prefect's central abstraction — orchestration is the management of states.

Run states

Every run moves through states. These are the words you'll read in the UI and in prefect flow-runs ls.

StateMeans
ScheduledPrefect intends to run it, at or after a certain time. Not started yet.
PendingPicked up, preparing to run (a brief transitional state).
RunningCurrently executing.
CompletedFinished successfully. The happy path.
CachedFinished successfully by reusing a stored result instead of running — a cache hit. A completed state, just one your code never executed for.
FailedRaised an exception the code didn't handle. A known, captured failure.
CrashedThe infrastructure died (process killed, OOM, power loss) — Prefect lost contact, distinct from a clean Failed.
PausedDeliberately suspended, often awaiting human input.
CancelledStopped on request before finishing.

Why it matters The difference between Failed and Crashed is the whole point of orchestration: Failed means your code hit an error it surfaced; Crashed means the machine went away. You debug them in completely different places.

Production vocabulary

TermMeaning
DeploymentA flow packaged with how/when to run it (schedule, parameters, name). Turns a script you run by hand into something Prefect triggers. See the deployments cheatsheet.
serve()flow.serve(name=…) — registers a deployment and runs a long-lived local process that fires & executes its scheduled runs. The local-first deployment path (no work pools).
ScheduleA rule attached to a deployment that creates Scheduled runs automatically — cron= (clock-based) or interval= (every N seconds).
State-change hookA function a flow/task calls when it enters a state — on_failure, on_completion, on_crashed, … Runs in the run's process. Used for per-flow alerting. See the alerts cheatsheet.
AutomationA server-side rule: a trigger (e.g. a flow run enters Failed/Crashed) → an action (e.g. send a notification). The server runs it, so it catches crashes a dead process can't report, and one rule covers every deployment.
Notification blockA reusable connection to an alert channel (SlackWebhook, CustomWebhookNotificationBlock, …). Save once, .load() + .notify(body, subject) from a hook or Automation.
RetryAutomatic re-execution of a failed task/flow, configured with retries= and retry_delay_seconds=.
CacheReusing a prior task result instead of recomputing. A hit yields a Cached state. Configured with cache_policy=. See the caching cheatsheet.
Cache key / policyA cache policy (e.g. INPUTS, TASK_SOURCE, DEFAULT) computes a cache key for each call. Same key as before = cache hit. Combine policies with +.
ResultThe return value of a task/flow run, which Prefect can persist (to disk or object storage) so it survives the process — the mechanism caching relies on.
Result storageWhere persisted results are written, set with result_storage=. Local by default; can be a remote storage block. Holds the return value, not your large artifacts.
Storage blockA saved, reusable connection to a storage backend (e.g. S3Bucket from prefect-aws, which also speaks S3-compatible stores like Tencent COS). Save once with .save(name), reload with .load(name). See the results & storage cheatsheet.
Work poolA typed queue that scheduled runs land in, waiting to be executed. Types: process (local subprocess), docker, kubernetes, … Create with prefect work-pool create <n> --type process.
WorkerA process (prefect worker start --pool <n>) that polls a work pool and executes the runs it finds. The "where" half of a deployment, separate from the "what". See the work pools cheatsheet.
deploy()flow.from_source(...).deploy(work_pool_name=…) — registers a deployment against a work pool and exits. The worker-based alternative to serve(); survives closed terminals and scales out.
Task runnerHow a flow's own tasks spread out — set with @flow(task_runner=…), acts only on .submit()/.map()ed tasks. ThreadPoolTaskRunner (default, concurrency for I/O) vs ProcessPoolTaskRunner (true multi-core parallelism for CPU). A different axis from work pools (which spread whole runs). See the task runners cheatsheet.
GILCPython's Global Interpreter Lock: only one thread runs Python bytecode at a time. Why threads help I/O (a waiting task releases it) but not CPU work (a computing task holds it) — for CPU parallelism you need separate processes.
Concurrency limitA cap on how many runs (or task acquisitions) may be active at once. Prefect counts runs — never cores or subprocess threads — so you translate cores into a run count yourself. See the concurrency cheatsheet.
Pool / queue limitA concurrency limit on a work pool (or queue), set with work-pool set-concurrency-limit. Fleet-global: shared by every worker polling it — a cap of 4 = 4 across the whole fleet, not 4 per machine.
Work queueA subdivision of a work pool with its own priority and optional concurrency limit. Workers can poll specific queues — e.g. route light vs heavy jobs differently within one pool.
Worker --limitworker start --limit N — caps concurrent whole flow runs on that one worker. Per-machine, but blunt: a light task waits behind a heavy one.
Global Concurrency Limit (GCL)A named, server-coordinated limit acquired in flow code via with concurrency(name, occupy=N). Name it per worker (f"encode:{WORKER_ID}") for a per-machine cap. Create it first with prefect gcl create — it is not auto-created.
occupyHow many slots one concurrency() acquisition consumes. Lets a heavy run claim several slots: with limit = cores and occupy = threads-per-encode, a machine never runs more threads than it has cores.
