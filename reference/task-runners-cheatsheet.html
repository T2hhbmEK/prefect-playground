Task runners

How the tasks within one flow run spread out. Thread vs process, copy-paste ready.

Two axes — don't confuse them

Task runnerWork pool + worker
Distributestask runs within one flow runwhole flow runs
Set inflow code: @flow(task_runner=…)deployment: work_pool_name=
Scalesone run's tasks across coresmany runs across machines

See the work pools cheatsheet for the other axis. They compose.

The rule that bites first

Only .submit() / .map() use the runner Plain task calls run sequentially regardless of task_runner. To get concurrency you must submit work: my_task.map(items) or my_task.submit(x).

The two built-in runners

RunnerGivesBest for
ThreadPoolTaskRunner (default)Concurrency (one process, many threads)I/O-bound: network, disk, DB waits
ProcessPoolTaskRunnerTrue parallelism (many processes, many cores)CPU-bound & heavy enough to amortize overhead

ConcurrentTaskRunner also exists (async-focused concurrency). Cluster runners DaskTaskRunner / RayTaskRunner (separate installs) are out of scope for now.

Usage

from prefect import flow, task
from prefect.task_runners import ThreadPoolTaskRunner, ProcessPoolTaskRunner

@flow(task_runner=ThreadPoolTaskRunner(max_workers=8))   # I/O-bound
def fetch_all():
    return fetch.map(sources).result()

@flow(task_runner=ProcessPoolTaskRunner(max_workers=4))  # heavy CPU
def crunch_all():
    return crunch.map(chunks).result()

Why threads don't speed up CPU work

CPython's GIL lets only one thread run Python bytecode at a time. A waiting task releases the GIL (so threads overlap I/O), but a computing task holds it (so threads take turns). Separate processes each have their own interpreter and core.
