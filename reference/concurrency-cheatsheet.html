Concurrency & CPU saturation

Prefect counts runs, not cores. Cap a heterogeneous fleet without oversubscribing. Copy-paste ready.

The reframe

Count runs, not cores Every Prefect limit counts runs (or task acquisitions) — never cores, and never the threads inside a subprocess (ffmpeg/SVT-AV1). Prefect can't see them, so it won't protect your CPU on its own. You compute cores ÷ threads-per-encode and hand it the number.

Three scopes — pick by what you're capping

Per-machine recipe (process pool, all local)

# 0. PIN THE ENCODER — or no limit can help you.
#    ffmpeg -threads 4 · SVT-AV1 --lp 4 · aomenc --threads=4

# 1. One GCL per machine, sized to its cores. Create it FIRST —
#    concurrency() does not auto-create limits.
uv run prefect gcl create encode:big-cvm-1 --limit 16   # 16-core box
uv run prefect gcl create encode:small-1   --limit 4    # 4-core box

# 2. Start each worker carrying its identity
WORKER_ID=big-cvm-1 uv run prefect worker start --pool encode --limit 8

# 3. In the flow, gate the subprocess by its thread cost
from prefect.concurrency.sync import concurrency
import os, subprocess
wid = os.environ["WORKER_ID"]
with concurrency(f"encode:{wid}", occupy=4):     # 4 threads → 4 slots
    subprocess.run(["ffmpeg", "-threads", "4", ...], check=True)

The formula limit = the machine's core budget · occupy = each encode's thread cost. A 16-core box admits ⌊16/4⌋ = 4 concurrent 4-thread encodes. Same flow code on every machine; only the limit number differs per box.

GCL CLI

CommandUse

gcl inspect <name>Limit config + live occupancy.
gcl update <name> --limit NResize a machine's budget (e.g. after adding cores).
work-pool set-concurrency-limit <n> <N>The fleet-global pool cap — different tool, different scope.

concurrency() is sync; use from prefect.concurrency.asyncio import concurrency + async with in async flows. Acquiring without releasing (a pure cap, no wrapping) is what rate_limit() is for.

Gotchas

Things that bite
  • Encoder ignores your limit → it auto-detected all cores. Pin -threads/--lp; occupy only describes a cost you actually enforce.
  • Pool limit ≠ per-machine → a pool cap of 4 means 4 across the whole fleet, not per box. Use a per-worker GCL.
  • Limit doesn't exist → concurrency() won't create it; gcl create first or the run errors.
  • Blocked run still holds a worker slot → a run parked on concurrency() is already a live process. Size --limit sanely so you don't spawn far more than can run.

Beyond Prefect (the boundary)

Prefect orchestrates (what/when/retries); it is not a resource scheduler. For a hard CPU cap regardless of the encoder, wrap the subprocess in cgroups/taskset (OS layer). When static per-machine limits start stranding cores on a large heterogeneous fleet, that's the signal to move the encode pool onto Kubernetes and let it pack jobs by CPU request — out of scope for now.
