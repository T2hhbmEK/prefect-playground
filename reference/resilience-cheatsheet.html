Retries & Timeouts

Every knob that keeps a transient blip from sinking a run — in one place, copy-paste ready.

The decorator arguments

ArgumentWhat it does
retries=NRe-run the task/flow up to N more times if it raises. Total attempts = N + 1.
retry_delay_seconds=SWait S seconds between attempts. A single number = fixed delay.
retry_delay_seconds=[1,2,4,8]A list = a different delay before each attempt.
retry_delay_seconds=exponential_backoff(backoff_factor=2)Auto-grow the delay: 2, 4, 8, 16… Import from prefect.tasks.
retry_condition_fn=fnA callable that decides per failure whether to retry. Return False to give up immediately.
timeout_seconds=TCancel the run if it exceeds T seconds. Marks it Failed; retries (if any) still apply.

Fixed-delay retry

from prefect import task

@task(retries=3, retry_delay_seconds=2)
def fetch(url):
    ...  # 4 total attempts, 2s apart

Exponential backoff

from prefect import task
from prefect.tasks import exponential_backoff

@task(retries=4, retry_delay_seconds=exponential_backoff(backoff_factor=2))
def fetch(url):
    ...  # waits 2s, 4s, 8s, 16s

Why backoff Hammering a struggling service every 2 seconds can keep it down. Backoff gives it room to recover — and is the default expectation of most APIs' rate limits.

Don't retry permanent errors

import httpx
from prefect import task

def retry_handler(task, task_run, state) -> bool:
    try:
        state.result()
    except httpx.HTTPStatusError as exc:
        # 401/404 won't fix themselves — don't waste attempts
        return exc.response.status_code not in (401, 404)
    except httpx.ConnectError:
        return False
    return True

@task(retries=3, retry_condition_fn=retry_handler)
def api_call(url):
    httpx.get(url).raise_for_status()

Timeouts

from prefect import flow

@flow(timeout_seconds=30)
def pipeline():
    ...  # killed if it runs longer than 30s
Caveat — sync tasks A timeout_seconds on a synchronous, blocking task (e.g. time.sleep) running under the default ThreadPoolTaskRunner may not interrupt mid-call — it only triggers once control returns. Flow-level timeouts and async tasks are reliable. (Out of scope now; revisit with task runners.)

Mental model

Failure kindRight tool
Transient (network blip, 503, brief lock)retries + delay — it'll probably work next time.
Permanent (bad auth, 404, bad input)retry_condition_fn returns False — fail fast, retries are wasted.
Hung / never returnstimeout_seconds — cap it so it can't run forever.
